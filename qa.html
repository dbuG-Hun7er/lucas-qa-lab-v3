Quality Assurance

QA não é botão bonito. É raciocínio, risco, evidência e decisão.

Esta página é meu mapa de QA: o que eu estudo, como eu penso e como conecto testes manuais, API, automação, performance, mobile, BDD e evidências em uma visão profissional.

Minha definição prática

QA é a área que ajuda o time a entregar software com mais confiança. Não é só encontrar bug no final. É participar da construção da qualidade desde requisito, refinamento, desenvolvimento, teste, entrega e aprendizado depois do deploy.

Um QA forte não pensa só “funcionou ou não funcionou”. Ele pensa: quem usa, qual risco, qual regra, qual exceção, qual evidência e qual impacto.

O que eu quero mostrar

  • Fundamento antes de ferramenta
  • Teste manual com intenção
  • API e automação com propósito
  • Bug report claro e útil
  • Risco, negócio e usuário

Mapa clicável de QA

Clique em cada card para abrir a explicação. Agora cada bloco tem conteúdo de verdade, não só palavra solta.

01 QA como leitura de risco Antes de qualquer ferramenta, o QA precisa entender onde mora o risco.

Risco é tudo que pode afetar usuário, negócio, dinheiro, segurança, reputação ou operação. Um QA forte não testa apenas o que foi pedido; ele tenta entender onde o produto pode falhar quando usado por pessoas reais.

  • Qual parte do fluxo gera mais impacto se quebrar?
  • O erro impede compra, login, cadastro, pagamento ou acesso?
  • Existe risco para dados do usuário?
  • O problema é visual, funcional, financeiro ou de segurança?
02 Requisito e critério de aceite Muitos bugs nascem antes do código: nascem em requisito mal explicado.

Quando uma regra é ambígua, cada pessoa entende de um jeito. O QA precisa questionar exemplos, exceções, mensagens, permissões e limites antes do desenvolvimento terminar.

  • O que deve acontecer no fluxo feliz?
  • O que deve acontecer quando falta dado?
  • Quem pode fazer essa ação?
  • Qual mensagem aparece quando falha?
03 Testes manuais profissionais Teste manual não é sair clicando. É testar com intenção, contexto e evidência.

Um teste manual bem feito tem cenário, pré-condição, massa de dados, passos, resultado esperado, resultado obtido e evidência. Ele pode ser mais valioso que uma automação mal feita.

  • Fluxo feliz
  • Fluxos alternativos
  • Cenários negativos
  • Bordas e exceções
04 Teste exploratório Explorar é investigar com método, não testar no improviso.

O teste exploratório é útil quando a funcionalidade é nova, quando o requisito está fraco ou quando o QA quer encontrar riscos fora do roteiro. Ele mistura experiência, curiosidade e hipóteses.

  • Tentar caminhos incomuns
  • Misturar ações
  • Voltar tela
  • Forçar limite de campo
05 Bug report que ajuda o time Bug report bom reduz atrito e acelera correção.

Um bug mal escrito gera discussão. Um bug bem escrito mostra o problema, o impacto, o ambiente e o caminho para reproduzir. Ele não acusa; ele prova.

  • Título claro
  • Ambiente
  • Passos para reproduzir
  • Resultado esperado e obtido
  • Evidência
  • Impacto
06 API Testing API é onde muita regra aparece antes de chegar na tela.

Testar API é validar contrato, payload, status code, autenticação, autorização, mensagens de erro e comportamento com dados inválidos. Um QA que entende API investiga melhor.

  • Status code
  • Payload
  • Campos obrigatórios
  • Token
  • Permissão
  • Erro
07 Automação com propósito Automação existe para proteger valor, não para enfeitar currículo.

Automatizar tudo é erro. O certo é automatizar fluxo repetitivo, crítico e estável. Automação boa é legível, confiável, fácil de manter e útil para o time.

  • Fluxos críticos
  • Regressão
  • Dados controlados
  • Menos flaky test
08 Cypress na prática Cypress entra como ferramenta para validar fluxo web com clareza.

Com Cypress eu posso trabalhar E2E, Page Objects, fixtures, comandos customizados, cy.intercept e validações visuais/funcionais. Mas Cypress é meio, não fim.

  • Page Objects
  • App Actions
  • Fixtures
  • cy.intercept()
  • Comandos customizados
09 BDD e comunicação BDD não é só Given/When/Then. É alinhamento.

BDD ajuda QA, dev e negócio a falarem a mesma língua. O valor está em transformar regra em exemplo claro antes de virar bug.

  • Cenário legível
  • Regra clara
  • Exemplo concreto
  • Menos ambiguidade
10 Performance Performance é experiência, custo e estabilidade.

Teste de performance observa tempo de resposta, carga, ramp-up, gargalo, erro e degradação. Ferramenta como k6 ou JMeter ajuda, mas o valor está na análise.

  • Tempo de resposta
  • Throughput
  • Erro
  • Gargalo
  • Carga
11 Mobile Testing Mobile quebra por contexto: rede, teclado, tela, permissão e estado do app.

Testar mobile exige pensar em fluxo real: login, busca, checkout, endereço, carrinho, permissões, rotação, app em background e conexão ruim.

  • Appium
  • WebdriverIO
  • Teclado
  • Permissões
  • Rede instável
