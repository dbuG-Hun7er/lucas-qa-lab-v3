Quality Assurance

QA não é só testar. É reduzir risco antes que ele vire prejuízo.

Minha visão de QA nasce da prática: observar falhas, entender impacto, documentar evidências e ajudar o time a entregar com mais confiança. Ferramenta é importante, mas fundamento vem antes: requisito, risco, usuário, regra de negócio, dados, API, fluxo e comunicação.

QA como leitura de risco

Antes de clicar, o QA precisa perguntar: o que pode quebrar, quem será afetado, qual o impacto e como eu provo o problema?

QA como proteção de valor

Um bug importante não é só um erro visual. É algo que afeta usuário, negócio, dinheiro, segurança, operação ou confiança.

QA como comunicação

Teste bom não termina no achado. Termina em evidência clara, bug report útil, conversa objetiva e decisão melhor para o time.

Mapa completo de QA

1. O que é QA na prática

QA é a área que ajuda o time a entregar software com mais confiança. Não é apenas encontrar bug no final. É participar da construção da qualidade desde o começo: requisito, refinamento, critérios de aceite, fluxo, risco, teste, evidência, automação e aprendizado depois da entrega.

Na prática, um QA forte observa o produto como sistema. Ele pensa no que o usuário quer fazer, no que o negócio espera, no que o sistema promete e no que pode quebrar quando as partes se encontram.

  • Questionar requisito ambíguo.
  • Mapear riscos antes do desenvolvimento terminar.
  • Validar regras de negócio em cenários reais.
  • Documentar evidências para apoiar decisões.
  • Conversar com dev, produto e negócio sem criar guerra.
2. Mentalidade de tester

A mentalidade de tester é a capacidade de desconfiar com método. Não é ser negativo. É olhar para o sistema e perguntar: “e se o usuário fizer diferente?”, “e se o dado vier errado?”, “e se a API responder incompleta?”, “e se a regra não cobrir essa exceção?”.

Essa mentalidade é o que separa alguém que apenas executa roteiro de alguém que realmente protege qualidade.

  • Não aceitar fluxo feliz como prova total.
  • Procurar exceções, bordas e comportamentos inesperados.
  • Entender o motivo do teste, não só o passo.
  • Transformar dúvida em cenário testável.
3. Testes manuais bem feitos

Teste manual não é sair clicando. Teste manual bem feito exige planejamento, intenção e observação. Ele começa entendendo o requisito e termina com evidências claras do que foi validado.

Um bom teste manual cobre fluxo feliz, fluxo alternativo, cenário negativo, bordas, permissões, mensagens, dados obrigatórios, comportamento visual e impacto da regra.

  • Cenário: situação que será validada.
  • Caso de teste: passo a passo para executar a validação.
  • Massa de dados: dados usados no teste.
  • Resultado esperado: como o sistema deveria se comportar.
  • Resultado obtido: como o sistema realmente se comportou.
4. Teste exploratório

Teste exploratório não é bagunça. É investigação guiada por risco. O QA usa conhecimento do sistema, curiosidade e hipóteses para encontrar problemas que o roteiro tradicional talvez não cubra.

Ele é muito útil quando o requisito é novo, quando há pouca documentação, quando uma funcionalidade parece frágil ou quando o QA quer entender comportamento real do produto.

  • Explorar combinações de ações.
  • Testar limites de campos.
  • Forçar caminhos incomuns.
  • Observar mensagens, logs, travamentos e inconsistências.
5. Bug report profissional

Bug report bom é aquele que o time consegue entender e reproduzir. Ele não pode ser emocional, confuso ou incompleto. Precisa mostrar o problema com clareza.

  • Título: direto, dizendo onde acontece e qual é o erro.
  • Ambiente: navegador, app, versão, dispositivo ou massa usada.
  • Passos: caminho exato para reproduzir.
  • Resultado esperado: comportamento correto.
  • Resultado obtido: comportamento atual.
  • Evidência: print, vídeo, log, payload ou resposta da API.
  • Impacto: por que o bug importa.
6. Testes de API

API Testing valida o sistema por trás da tela. Muitas falhas importantes não aparecem visualmente primeiro. Elas aparecem em payload errado, status incorreto, autenticação falha, autorização quebrada, contrato inconsistente ou regra mal aplicada.

Um QA que entende API consegue investigar melhor, falar com mais segurança com devs e validar comportamento sem depender apenas do front-end.

  • Status code correto.
  • Payload esperado.
  • Campos obrigatórios.
  • Mensagens de erro.
  • Token ausente ou inválido.
  • Usuário sem permissão.
  • Contrato entre front e back-end.
7. Automação com propósito

Automação não é para mostrar que sabe escrever script. Automação existe para reduzir retrabalho, proteger fluxo crítico e aumentar confiança nas mudanças.

Um projeto de automação precisa ser legível, estável e fácil de manter. Se o teste quebra por qualquer timing ou depende de massa instável, ele deixa de ajudar e começa a atrapalhar.

  • Automatizar fluxos repetitivos e críticos.
  • Evitar automatizar coisa instável demais cedo.
  • Usar Page Objects ou uma estrutura equivalente quando fizer sentido.
  • Separar dados, ações e validações.
  • Tratar flaky tests como problema real.
8. Cypress no meu caminho

Cypress entra na minha jornada como ferramenta principal para automação E2E web. Ele permite validar fluxo de usuário, interceptar requisições, usar fixtures, criar comandos customizados e organizar testes com boa leitura.

Mas a ideia não é virar dependente de Cypress. A ideia é aprender automação com base sólida: intenção do teste, dado controlado, isolamento quando possível e validação que faça sentido.

9. BDD e Cucumber

BDD ajuda a transformar regra em conversa. O valor não está em escrever “Given, When, Then” bonito. O valor está em alinhar entendimento antes do código e antes do bug.

Quando bem usado, BDD aproxima QA, dev e negócio. Quando mal usado, vira só uma camada a mais para manter.

10. Performance

Performance testa comportamento sob carga e pressão. Não é só perguntar quantos usuários o sistema aguenta. É observar tempo de resposta, erro, estabilidade, gargalo, ramp-up, duração e degradação.

Ferramentas como JMeter e k6 ajudam a simular cenários, mas o valor está na análise: onde degradou, por que degradou e qual risco isso gera.

11. Mobile Testing

Mobile exige atenção especial porque o ambiente muda muito: tamanho de tela, teclado, permissões, rede, dispositivo lento, sistema operacional, gestos, rotação e estado do app.

Um teste mobile bom precisa considerar fluxo real: login, busca, carrinho, endereço, checkout, background, retorno ao app e mensagens de erro.

12. QA Júnior: como eu quero evoluir

Minha meta como QA Júnior é construir base real, não só lista de ferramentas. Quero fortalecer testes manuais, API, automação, SQL, Git, evidências, bug report, performance e segurança básica.

O caminho é simples, mas exige consistência: estudar, praticar, publicar projetos, melhorar documentação, revisar erros e evoluir com humildade técnica.
