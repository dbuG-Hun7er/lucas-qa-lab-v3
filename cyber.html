Cybersecurity

Cyber para QA: quando bug vira risco, dado exposto ou porta aberta.

Minha jornada em segurança está em construção, mas ela tem um papel claro: tornar meu olhar de QA mais completo. Um sistema precisa funcionar, mas também precisa proteger dados, respeitar permissões e resistir ao uso indevido.

Minha visão de Cyber dentro do QA

Cybersecurity aqui não entra como fantasia de hacker. Entra como qualidade aplicada ao risco. O QA que entende o básico de segurança testa melhor API, autenticação, autorização, dados, logs, mensagens de erro e abuso de fluxo.

O objetivo é simples: não validar apenas o usuário correto fazendo a coisa certa. Também pensar no usuário tentando fazer o que não deveria.

Cyber mindset

  • O que pode ser acessado indevidamente?
  • Qual dado pode vazar?
  • Qual regra pode ser burlada?
  • O back-end protege ou só o front esconde?
  • Como reportar sem expor informação sensível?

Mapa clicável de Cyber para QA

Clique em cada card para abrir a explicação. Cada item mostra como eu conecto segurança com testes.

01 Risco Segurança começa perguntando o que pode dar errado.

Risco é quando uma falha deixa de ser apenas bug e passa a ter impacto real: dados expostos, acesso indevido, fraude, abuso de fluxo ou perda de confiança.

  • Dado exposto
  • Permissão errada
  • Fluxo burlado
  • Impacto financeiro
  • Impacto reputacional
02 API Security API é porta de entrada do sistema.

Não basta API retornar 200. É preciso validar contrato, autenticação, autorização, payload inválido, erro, limite e exposição indevida de informação.

  • Token ausente
  • Payload manipulado
  • ID alterado
  • Campo sensível
  • Status incorreto
03 Autenticação Quem é o usuário?

Autenticação valida identidade. Como QA, eu preciso pensar em login inválido, sessão expirada, logout, token vencido, tentativa sem login e comportamento em múltiplas sessões.

  • Login
  • Token
  • Sessão
  • Logout
  • MFA quando existir
04 Autorização O que esse usuário pode fazer?

Autorização é uma das partes mais críticas. Um usuário autenticado não pode acessar tudo. Muita falha séria acontece quando o front esconde botão, mas o back libera endpoint.

  • Admin vs usuário
  • Acesso por ID
  • Endpoint protegido
  • Permissão por perfil
05 OWASP como mapa mental Não é decorar lista. É aprender a perguntar melhor.

OWASP ajuda a organizar riscos comuns: controle de acesso, falhas criptográficas, injeção, configuração insegura, autenticação, integridade e exposição de dados.

  • Controle de acesso
  • Injeção
  • Configuração
  • Dados sensíveis
  • Autenticação
06 Validação de entrada Campo aberto demais vira risco.

Entrada de usuário precisa ser validada. Campo, query string, upload, payload e parâmetro podem receber dados inválidos, grandes demais, especiais ou maliciosos.

  • Campo vazio
  • Tipo errado
  • Texto enorme
  • Caracteres especiais
  • Payload inesperado
07 Dados sensíveis Sistema bom não fala mais do que precisa.

Um sistema pode funcionar e ainda vazar informação. QA precisa observar tela, resposta da API, mensagens de erro, logs e arquivos para ver se dados sensíveis aparecem indevidamente.

  • Senha
  • Token
  • CPF
  • E-mail
  • Dados internos
08 Logs Log ajuda investigação, mas também pode vazar segredo.

Logs são úteis, mas precisam ser seguros. Não devem expor senha, token, segredo, dado pessoal ou informação interna demais.

  • Token em log
  • Senha em erro
  • Stack trace
  • Dados pessoais
09 Abuso de fluxo Usuário malicioso não segue o caminho feliz.

Ele volta tela, repete ação, altera ID, chama endpoint direto, muda payload, tenta pular etapa e força limite. QA com visão cyber testa além do usuário comportado.

  • Repetir requisição
  • Pular etapa
  • Alterar ID
  • Forçar limite
  • Manipular payload
10 Mobile Security No mobile, o risco também mora no dispositivo.

App pode guardar sessão, cache, token e dados locais. Também sofre com rede instável, permissões, app em segundo plano e comportamento de logout.

  • Cache
  • Sessão
  • Token
  • Permissões
  • Background
11 Bug report de segurança Falha sensível exige comunicação responsável.

Bug de segurança não deve ser espalhado de qualquer jeito. O reporte precisa ter evidência, impacto, passos e cuidado para não expor dado real ou detalhe sensível desnecessário.

  • Evidência controlada
  • Massa fictícia
  • Impacto claro
  • Canal correto
12 Minha trilha Cyber para QA Ser honesto: estou construindo base para ser QA mais completo.

Meu objetivo não é fingir senioridade em segurança. É aprender o suficiente para pensar melhor em risco, API, auth, dados e uso malicioso do sistema.

  • Fundamentos web
  • API
  • OWASP
  • Redes
  • Ética
