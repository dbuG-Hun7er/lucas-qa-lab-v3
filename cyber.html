Cybersecurity

Segurança é qualidade quando a falha deixa de ser apenas um bug e passa a ser uma porta aberta.

Minha jornada em cyber ainda está em construção, mas ela faz sentido dentro do QA: um sistema bom não precisa apenas funcionar. Ele precisa proteger dados, respeitar permissões, resistir a abuso e manter a confiança do usuário.

Cyber como extensão de QA

QA olha qualidade. Cyber olha risco de segurança. Quando as duas visões se encontram, o teste fica mais forte.

Não vender senioridade falsa

Minha proposta é ser honesto: estou construindo base em segurança para pensar melhor como QA.

Foco prático

API, autenticação, autorização, dados, logs, validação de entrada, abuso de fluxo e OWASP como mapa mental.

Mapa de Cyber para QA

Clique em cada bloco para abrir a explicação detalhada.

1. Risco

Risco é a chance de uma falha causar impacto real. Em QA, muitas vezes pensamos em bug funcional. Em cyber, pensamos no que essa falha permite: acesso indevido, vazamento, fraude, abuso, escalada de permissão ou quebra de confiança.

  • Qual dado pode ser exposto?
  • Quem pode acessar o que não deveria?
  • O usuário consegue burlar uma regra?
  • O erro causa impacto financeiro ou reputacional?
2. Autenticação

Autenticação responde: quem é o usuário? Login, senha, token, sessão e MFA fazem parte desse universo.

Como QA, eu preciso pensar se o sistema lida bem com token expirado, senha errada, sessão vencida, logout, login simultâneo e tentativa de acessar recurso sem estar autenticado.

3. Autorização

Autorização responde: o que esse usuário pode fazer? Esse ponto é crítico. Um usuário comum não pode acessar recurso de admin. Um cliente não pode ver dados de outro cliente.

Muito problema sério nasce quando o sistema autentica corretamente, mas autoriza errado.

  • Usuário comum acessando rota administrativa.
  • Alteração de ID na URL para ver dados de outro usuário.
  • Botão escondido no front, mas endpoint liberado no back-end.
4. API Security

APIs são portas de entrada do sistema. Não basta testar status 200. É preciso testar o que acontece quando o payload vem errado, quando o token está ausente, quando o usuário não tem permissão ou quando o dado é manipulado.

  • Endpoint aceita campo que não deveria?
  • Retorna dados sensíveis demais?
  • Permite alterar recurso de outro usuário?
  • Valida limite, tipo e obrigatoriedade?
5. Validação de entrada

Validação de entrada é uma barreira básica. Campo de texto, número, e-mail, CPF, senha, upload, query string e payload precisam rejeitar dados inválidos ou perigosos.

Para QA, isso vira cenário de teste: campo vazio, formato inválido, excesso de caracteres, caracteres especiais, scripts, payload inesperado e manipulação manual da requisição.

6. Dados sensíveis

Um sistema pode funcionar e ainda assim vazar informação. QA precisa observar se a aplicação mostra dados demais na tela, na resposta da API, nos logs ou em mensagens de erro.

  • Senha nunca deve aparecer.
  • Token não deve vazar em log.
  • Dados pessoais precisam de cuidado.
  • Mensagem de erro não deve revelar detalhe interno desnecessário.
7. Logs

Logs ajudam muito na investigação, mas também podem virar risco. O sistema precisa registrar informação útil sem expor senha, token, dados pessoais ou segredo de ambiente.

Como QA, eu posso observar mensagens, console, retorno de erro, logs de teste e evidências para perceber se existe exposição indevida.

8. Abuso de fluxo

Usuário malicioso não segue o caminho feliz. Ele repete ação, altera ID, volta tela, chama endpoint direto, manipula payload, tenta pular etapa e força limite.

Cyber para QA é pensar nesse usuário também. Não para virar hacker de filme, mas para testar se o sistema resiste ao uso indevido.

9. OWASP como mapa mental

OWASP ajuda a organizar riscos comuns de aplicações web e APIs. Para mim, como QA, serve como mapa mental para pensar em autenticação, autorização, exposição de dados, validação de entrada, configuração insegura e falhas de controle de acesso.

A ideia não é decorar lista. É aprender a fazer perguntas melhores durante o teste.

10. Segurança em testes mobile

No mobile, além do fluxo normal, existe o cuidado com sessão, token, cache, dados salvos no dispositivo, permissões, prints, logout e comportamento em rede instável.

Um app pode parecer correto visualmente, mas manter sessão indevida ou expor dado sensível de forma errada.

11. Segurança e bug report

Bug de segurança precisa de cuidado na comunicação. Não é para expor publicamente, nem espalhar detalhe sensível sem necessidade. O ideal é registrar de forma responsável, com evidência suficiente e controle de acesso.

  • Descrever impacto sem sensacionalismo.
  • Evitar vazar dados reais em print.
  • Usar massa fictícia quando possível.
  • Comunicar prioridade com clareza.
12. Minha trilha cyber dentro do QA

Meu objetivo não é fingir que sou especialista sênior em segurança. Meu objetivo é construir uma base real para ser um QA mais completo.

A trilha faz sentido assim: fundamentos de rede, web, API, autenticação, autorização, OWASP, testes de segurança básicos, documentação de evidência e ética profissional.
