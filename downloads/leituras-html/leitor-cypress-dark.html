Testes Web End-to-End com Cypress

Prefácio

Cypress é um framework de automação de testes em JavaScript criado com o único propósito de realizar testes de frontend. O Cypress se destaca por reinventar a forma como os testes são realizados, especialmente para a web moderna. Diferentemente de outros frameworks de teste, como o Selenium WebDriver, o Cypress é muito mais rápido pois roda no navegador e também tem uma curva de aprendizado menor em comparação com outros frameworks de teste.

Desenvolvedores que trabalham com aplicações frontend poderão colocar seus conhecimentos em prática com este guia prático e desenvolver suas habilidades em testes end-to-end. Este livro adota uma abordagem prática de implementação e metodologias associadas que farão você estar em funcionamento e produtivo em pouco tempo.

Para quem é este livro

O livro é destinado a profissionais de teste, testadores de software e web, e desenvolvedores web que dominam JavaScript e podem ou não estar familiarizados com os conceitos de automação de testes. Os três primeiros capítulos oferecem um guia introdutório rápido que ajudará você a se familiarizar com o funcionamento do Cypress e como começar caso seja um completo iniciante no Cypress. Se você é um testador de Selenium que deseja migrar para o Cypress e descobrir suas capacidades, este livro será muito útil. É necessário um bom entendimento de testes web e JavaScript.

Capítulo 1 — Instalando e Configurando o Cypress

O Cypress é um framework de automação de testes end-to-end construído e projetado para aplicações web modernas. Ele foca em eliminar inconsistências nos testes, garantindo que você possa escrever, depurar e executar testes no navegador sem precisar de configurações adicionais ou pacotes extras. O Cypress funciona como uma aplicação independente e pode ser instalado no macOS, Unix/Linux e Windows, seja usando aplicações de interface gráfica ou ferramentas de linha de comando. O Cypress foi principalmente construído para desenvolvedores que escrevem suas aplicações usando JavaScript, pois pode ser usado para testar todas as aplicações que rodam em um navegador. Neste capítulo, vamos abordar os seguintes tópicos:

  • Instalando o Cypress no Windows
  • Instalando o Cypress no macOS
  • Instalando o Cypress via download direto
  • Abrindo o test runner do Cypress
  • Alternando navegadores no Cypress

  • Executando testes Cypress

Ao final deste capítulo, você entenderá como configurar corretamente o Cypress tanto no Windows quanto no Mac e como executar testes Cypress. Você também entenderá como os scripts npm podem ser usados para automatizar o processo de execução de testes e abertura do test runner.

Requisitos Técnicos

O Cypress pode ser instalado como uma aplicação independente em seu computador e pode ser executado em uma máquina com pelo menos 2 GB de RAM que atenda a qualquer um dos seguintes requisitos de sistema operacional:

  • macOS 10.9 e acima (apenas 64-bit)
  • Linux Ubuntu 12.04 e acima, Fedora 21 e Debian 8 (apenas 64-bit)
  • Windows 7 e acima

Para usar o Cypress em um dos sistemas operacionais listados aqui, o Node.js 8 ou superior deve ser instalado primeiro. O Node.js é um ambiente de execução JavaScript que permite que o código JavaScript seja executado fora do navegador. A instalação do Node.js instala o npm, que nos permite instalar pacotes JavaScript de https://www.npmjs.com/. O npm é o gerenciador de pacotes padrão para Node.js, e os usuários podem usá-lo ou utilizar gerenciadores de pacotes de terceiros, como o Yarn. Nesta seção, instalaremos o Cypress tanto no macOS quanto no Windows.

Baixando e instalando o Node.js

Inicializando o projeto

Como boa prática, o Cypress é instalado no diretório onde o projeto está localizado; dessa forma, podemos ter certeza de que os testes Cypress pertencem ao projeto. Em nosso caso, criaremos uma pasta dentro de Documents e a chamaremos de cypress-tests, depois navegaremos até esse diretório ao instalar o Cypress. Podemos usar os seguintes comandos em um terminal Windows PowerShell para criar o diretório cypress-tests e navegar até ele:

$ cd mkdir cypress-tests

Depois que esses comandos forem executados com sucesso, lançaremos o PowerShell e navegaremos até o diretório que acabamos de criar com o seguinte comando:

$ cd .\Documents\cypress-tests

$ npm init –y

$ npm install cypress --save-dev

Após executar este comando, você deverá ver a instalação do Cypress e o progresso da instalação. Essa abordagem instala o Cypress como uma dependência de desenvolvimento para nosso projeto vazio.

Abrindo o Cypress

Instalar o Cypress é o primeiro passo na jornada de escrever testes end-to-end; agora, precisamos aprender como usar as ferramentas que o Cypress fornece para executar os testes tanto pela interface gráfica do usuário quanto pelos dashboards. Existem quatro maneiras de executar o executável do Cypress instalado em sua máquina. Após abrir o Cypress, você deverá ver o test runner do Cypress. Não importa de que forma você abra o Cypress, o dashboard do test runner que é apresentado a você é o mesmo. As seções a seguir detalham as diferentes formas de abrir e executar o Cypress.

Executando com Npx

O npx é usado para executar binários de pacotes npm e vem com todas as versões npm a partir da versão 5.2. O Npx também pode ser instalado usando npm do npmjs.com. Para executar o Cypress usando o npx, você precisa executar o seguinte comando:

npx cypress open

Executando com Yarn

Se o Cypress foi instalado usando o Yarn, você pode então abrir o Cypress usando o seguinte comando:

Yarn run cypress open

Executando com o caminho do node modules

O Cypress também pode ser executado referenciando o caminho raiz da instalação nos node modules. Isso pode ser feito tanto usando o caminho completo para o bin do node_modules onde o executável do Cypress está localizado, quanto usando o atalho de bin do npm.

Lançando o Cypress usando o caminho completo

Este método de lançamento do Cypress referencia o executável do Cypress instalado localizado em node_modules e funciona executando o executável para abrir o Cypress:

$ ./node_modules/.bin/cypress open

Lançando o Cypress usando o atalho

Assim como ao lançar o Cypress usando o caminho completo, este método lança o Cypress da mesma forma, mas em vez de referenciar o caminho completo, usa a variável bin do npm para localizar o local padrão da pasta bin do node_modules:

$(npm bin)/cypress open

Lançamento como aplicação desktop

Se você baixou sua aplicação como uma aplicação desktop, você pode abrir o Cypress navegando até o local da pasta Cypress descompactada e clicando no executável do Cypress que está presente na pasta.

Alternando navegadores

O Cypress vem com o Electron como navegador padrão na instalação, mas também pode integrar com outros navegadores compatíveis que contêm o projeto Chromium, com exceção do Firefox. Atualmente, o Cypress suporta navegadores Firefox, Chrome, Chromium e Edge. Ao lançar o Cypress, ele encontrará automaticamente todos os navegadores compatíveis na máquina em execução e você poderá alternar entre quaisquer navegadores a qualquer momento usando o test runner. Para alternar de um navegador para outro, você precisará clicar no botão do navegador no canto superior direito e escolher um navegador alternativo no link dropdown.

Os testes Cypress também podem ser executados ou abertos em diferentes navegadores usando a linha de comando, e isso pode ser conseguido especificando o navegador ao abrir o test runner do Cypress ou ao executar os testes Cypress. Todos os navegadores baseados em Chromium, Edge e Firefox podem ser lançados usando a linha de comando com o seguinte comando:

$ cypress run --browser {browser-name}

O browser-name especificado no comando pode ser Edge, Chrome ou Firefox. Para especificar o caminho do navegador que o Cypress deve lançar, você pode optar por executar o nome do navegador com o binário executável do navegador em vez do nome do navegador, como mostrado aqui:

$ cypress run --browser /path/to/binary/of/browser

Ser capaz de alternar navegadores no Cypress garante que um usuário possa executar seu conjunto de testes em diferentes dispositivos e verificar que a saída de diferentes navegadores é consistente ao longo do conjunto de testes. Alternar navegadores no Cypress também garante que a verificação dos testes possa ocorrer e que todos os elementos visíveis ou ações que podem ser realizadas em um navegador possam ser realizadas em outro.

Adicionando scripts npm

Os scripts npm permitem que você execute comandos a partir do arquivo package.json diretamente da linha de comando. O Cypress permite definir scripts npm que você pode usar para automatizar o processo de execução de testes, seja por meio da linha de comando ou por meio do test runner do Cypress. Ao executar comandos Cypress via linha de comando, você estará no modo headless, o que significa que os testes serão executados sem a necessidade de abrir um navegador visível.

Para adicionar scripts npm ao arquivo package.json, você precisará modificar a seção "scripts" do arquivo:

{
  "scripts": {
    "cypress:run": "cypress run",
    "cypress:open": "cypress open"
  }
}

Para executar qualquer um desses scripts, usamos npm run seguido pelo nome do script:

npm run cypress:run
npm run cypress:open

Por que escolher o Cypress?

O Cypress é um framework de testes end-to-end escrito por desenvolvedores, para desenvolvedores e engenheiros de Quality Assurance (QA). O Cypress foca em testar aplicações web e, como a única forma de automatizar a web é usando JavaScript, o Cypress só suporta o uso de JavaScript para escrever seus testes.

O Cypress foi especificamente escrito para equipes de frontend que utilizam JavaScript para o desenvolvimento de seus produtos, junto com equipes que precisam começar rapidamente com o processo de escrever testes unitários, de integração e end-to-end sem as complicadas complexidades de configurar corretamente um framework de testes.

O Cypress não é apenas amigável para iniciantes, mas também garante que tudo que um desenvolvedor ou engenheiro de QA precisa para começar seus testes já está empacotado no bundle que é baixado e instalado do site do Cypress. O Cypress vem com seu próprio navegador, um test runner e o Chai como framework de asserção.

Ter um bundle que contém tudo necessário para começar o processo de escrever testes significa que qualquer pessoa pode simplesmente começar com o trabalho de testes sem precisar saber sobre o processo de configuração de frameworks de asserção, test runners ou mesmo a adição de drivers de navegador, como no caso de usar o Selenium WebDriver.

O Cypress faz uso de JavaScript, o que o torna significativamente mais fácil para desenvolvedores JavaScript embarcarem e aprenderem rapidamente os conceitos do Cypress. A facilidade de embarque também garante que desenvolvedores e engenheiros de QA possam rapidamente se adaptar à escrita de testes usando Cypress.

O Cypress utiliza um driver universal que é atualmente compatível com Firefox, Edge, Chrome e a família de navegadores Chromium. Diferente do Selenium, que utiliza o WebDriver e interage com o Document Object Model (DOM) usando requisições de rede HTTP, o driver do Cypress trabalha diretamente no navegador sem a necessidade de fazer requisições de rede. A capacidade de executar dentro do navegador com os testes garante que o Cypress possa interpretar efetivamente os comandos sem introduzir timeouts quando os comandos são passados dos testes para o driver e depois para a aplicação em execução no navegador.

O framework Cypress roda no navegador, pois é arquiteturalmente diferente de qualquer outra ferramenta de automação de testes como o Selenium WebDriver. A capacidade do Cypress de rodar no navegador lhe dá uma vantagem competitiva sobre outras ferramentas de automação, pois vem com sequências de espera automáticas que de outra forma precisariam ser definidas nos testes. O Cypress, portanto, sabe quando esperar por um evento, como uma requisição de rede, que de outra forma precisaria ser especificado como uma espera explícita ou implícita em um teste orientado pelo Selenium.

Recapitulando – por que escolher o Cypress?

Nesta seção, aprendemos por que o Cypress é a escolha preferida quando se trata de testes de desenvolvimento web e o que o destaca de outros frameworks de testes, incluindo o Selenium WebDriver.

Comparando Cypress e Selenium WebDriver

É fácil cair na armadilha de fazer suposições de que o Cypress é um substituto para o Selenium WebDriver e que seu uso poderia tornar o Selenium WebDriver completamente obsoleto no mundo da automação de testes.

Driver do navegador

O Cypress utiliza um driver universal personalizado para todos os navegadores que suporta, enquanto o Selenium WebDriver utiliza drivers diferentes para cada um dos diferentes navegadores que suporta. A capacidade de usar um driver universal para todos os navegadores significa que, na instalação, podemos executar nossos testes em todos os navegadores suportados pelo Cypress sem a necessidade de instalar um driver externo.

Retries e waits

O Cypress vem com retries explícitos embutidos para procurar elementos no DOM e explicitamente aguarda que eventos aconteçam antes que um teste seja considerado como falho. O Cypress vem acoplado com eventos que determinam se as requisições precisam ser aguardadas antes que o navegador decida se elas falharam ou passaram. O Cypress é capaz de lidar com os waits e retries porque roda no navegador com os testes e é capaz de entender o estado dos testes a qualquer momento.

O Selenium, por outro lado, utiliza requisições HTTP para o WebDriver e é, portanto, difícil para o framework determinar se waits explícitos ou implícitos são necessários quando os testes estão sendo executados. Para resolver o problema, os usuários do Selenium têm que escrever os waits eles mesmos em situações onde os testes precisariam aguardar que uma requisição fosse concluída antes de prosseguir para a próxima etapa de execução.

Uso pretendido

O Cypress é construído para desenvolvedores JavaScript e engenheiros de QA que desejam configurar rapidamente um framework de automação e começar a testar suas aplicações web end-to-end, sem gastar muita largura de banda configurando frameworks de teste.

O Selenium WebDriver, por outro lado, é construído para testar qualquer coisa que rode na web. O Selenium foca em equipes de QA que querem testar todos os aspectos de suas aplicações web e não é limitado por fatores como compatibilidade de navegador ou um único test runner. O Selenium WebDriver dá ao usuário opções para estendê-lo com diferentes navegadores e plugins, e também suporta diferentes linguagens como Java, Python, Ruby, C#, JavaScript, Perl e PHP.

Arquitetura

O Cypress roda no navegador e isso lhe dá uma vantagem sobre ferramentas como o Selenium WebDriver. Rodar no navegador significa que o Cypress é significativamente mais rápido e pode interpretar comandos mais rapidamente em tempo de execução, pois não há serviços de terceiros interpretando os comandos em seu nome ou enviando requisições HTTP para o driver do navegador.

Compatibilidade entre navegadores

O Cypress não suporta atualmente todos os principais navegadores da forma que o Selenium WebDriver faz. O Cypress atualmente suporta navegadores construídos usando o projeto open source Chromium, Firefox, Edge e Electron (o navegador padrão no Cypress).

Trade-offs do Cypress

Como mencionado anteriormente, o Cypress é uma ferramenta de automação de testes focada em automação de testes end-to-end no navegador. Ser capaz de rodar no navegador significa que o Cypress pode interagir com os elementos do navegador melhor do que qualquer outra ferramenta, mas também significa que o Cypress tem trade-offs permanentes que não podem ser alterados devido à sua arquitetura.

Limitações de escopo: O Cypress funciona melhor quando usado como utilitário de automação para engenheiros de QA e desenvolvedores que estão escrevendo testes. O Cypress não suporta ferramentas para automação manual e não tem planos de integrar ferramentas de testes manuais no framework.

Limitações de ambiente: O Cypress roda no navegador e isso significa que a linguagem que suporta sempre será JavaScript. Para se conectar a um banco de dados ou servidor, só podemos usar os comandos Cypress de cy.exec(), cy.request() ou cy.task().

Múltiplos navegadores e múltiplas abas – limitações: O framework Cypress não suporta a capacidade de um teste controlar múltiplos navegadores quando em execução. O framework Cypress também não suporta a capacidade de interagir com mais de uma aba do navegador.

Limitações de controle de origem: O Cypress só suporta visitar URLs que são da mesma origem no mesmo teste.

A forma correta de utilizar cross-origin para executar testes

No seguinte teste, o usuário instrui o Cypress a primeiro navegar para o site do GitHub (https://github.com) e depois para https://docs.github.com/en (link de documentação) dos recursos do GitHub. Ambos os links pertencem à mesma origem, github.com, e, portanto, o Cypress não terá problemas para executar as requisições:

It('can navigate to code repository hosting service', () => {
    cy.visit('https://github.com');
    cy.visit('https://docs.github.com');
});

A forma incorreta de utilizar cross-origin para executar testes

Neste teste, o usuário instrui o Cypress a primeiro navegar para https://github.com, depois navegar para https://gitlab.com, que é de uma origem diferente da primeira URL. Isso levará a um erro sendo lançado quando o teste for executado:

It('can navigate to code repository hosting service', () => {
    cy.visit('https://github.com');
    cy.visit('https://gitlab.com');
})

Ações complementares de Cypress e Selenium

Podemos utilizar tanto o Cypress quanto o Selenium juntos em algumas instâncias raras, mas ainda alcançáveis, ao escrever nossos testes. Enquanto o Cypress tem a limitação de não poder controlar mais de uma aba do navegador, é possível configurar o Cypress para usar o Selenium para executar múltiplas abas.

Usando o Cypress para testes de frontend

O Cypress é construído para a web e é otimizado para rodar em navegadores. A arquitetura do Cypress permite que ele execute testes efetivamente, ao mesmo tempo em que supera os desafios do WebDriver. O Cypress também mira em engenheiros de QA e desenvolvedores que estão buscando escrever testes sem se preocupar com a infraestrutura subjacente.

Executando comandos Cypress

A utilização eficaz do framework Cypress requer que você tenha um entendimento do Cypress e como diferentes funcionalidades podem ser executadas usando a linha de comando. Os comandos Cypress permitem que os usuários do framework Cypress automatizem processos e também forneçam instruções específicas para o framework e os testes durante a inicialização e em tempo de execução.

Na maioria das instâncias, executar testes Cypress pela linha de comando é mais rápido do que executá-los usando o navegador. Isso ocorre porque executar testes pela linha de comando reduz o número de recursos necessários para executar um teste específico. A razão para isso é que os testes executados na linha de comando são normalmente headless, o que significa que menos recursos são alocados para executar os testes.

Nota importante O modo headed é quando os testes podem ser visualmente vistos rodando em um navegador, enquanto no modo headless, o processo de execução do teste não abre um navegador visível. Em vez disso, todos os testes são executados e exibidos na linha de comando.

Comandos globais e locais

Os comandos Cypress podem ser executados a partir de um diretório específico contendo uma instalação e código Cypress, ou executados a partir da instalação global do Cypress. Instalar o Cypress globalmente garante que os usuários possam executar o Cypress a partir de qualquer diretório no sistema operacional, enquanto com uma instalação local do Cypress, o Cypress só pode ser acessado a partir do único diretório em que foi instalado.

Executando comandos globais do Cypress

Para poder executar comandos Cypress globalmente, você precisa instalar o Cypress globalmente com o seguinte comando:

npm install cypress --global
ou (versão mais curta)
npm i -g cypress

Executando comandos locais do Cypress

Para executar comandos Cypress localmente, você precisará instalar o Cypress em seu diretório com o seguinte comando:

npm install cypress
ou (versão mais curta)
npm i cypress

Podemos integrar os comandos necessários ao ambiente de desenvolvimento definindo-os no arquivo package.json, na seção de scripts:

{
  "scripts": {
    "cypress:run": "cypress run",
    "cypress:open": "cypress open"
  }
}

Para executar scripts na linha de comando, você precisa invocar npm run, seguido pelo nome do comando:

npm run cypress:run // comando para executar testes no terminal
npm run cypress:open // comando para executar testes no cypress runner

Nota importante A equipe de desenvolvimento do Cypress recomenda instalar o Cypress por projeto e não usar a abordagem de instalação global. A instalação local fornece certas vantagens, como a capacidade dos usuários de atualizar rapidamente a dependência do Cypress e reduzir os problemas de dependência cíclica.

Entendendo comandos básicos do Cypress

Nesta seção, exploraremos vários comandos Cypress que podemos usar para executar nossos testes. Os comandos que exploraremos são os seguintes:

  • cypress run
  • cypress open

cypress run

O comando cypress run executa todos os testes em um conjunto Cypress de forma headless e executa os testes no navegador Electron por padrão. As opções de configuração mais comuns do Cypress incluem as seguintes:

cypress run --env <env-variable>

As variáveis de ambiente do Cypress são pares nome-valor dinâmicos que influenciam a forma como o Cypress executa os testes. Essas variáveis de ambiente são úteis quando há necessidade de executar os testes em múltiplos ambientes.

Por exemplo, no arquivo Todo-app.spec.js:

context('TODO MVC Application Tests', () => {
  beforeEach(() => {
    cy.visit(
      `${Cypress.env('TransferProtocol')}://todomvc.com/examples/react/#/`)
  });

E no package.json:

"scripts": {
  "cypress:run": "cypress run --env TransferProtocol='http'",
  "cypress:run:v2": "cypress run --env TransferProtocol='https'"
}

Nota importante HTTPS é o mesmo que HTTP, com a diferença de que HTTPS é mais seguro. Isso ocorre porque o processo de envio de requisições e recebimento de respostas é criptografado usando o protocolo de segurança TLS(SSL).

cypress run --browser <browser-name>

A linha de comando do Cypress tem a capacidade embutida de executar testes Cypress em diferentes navegadores que foram instalados no computador host. O Cypress tenta detectar automaticamente os navegadores instalados e pode executar os testes em chrome, chromium, edge, firefox ou electron.

"scripts": {
  "cypress:chrome": "cypress run --env TransferProtocol='http' --browser chrome",
  "cypress:firefox": "cypress run --env TransferProtocol='http' --browser firefox"
}

cypress run --config <configuration(s)-option>

O Cypress pode definir e sobrescrever configurações usando comandos no terminal. As opções de configuração podem incluir especificar um viewportHeight e ViewportWidth alternativos, timeouts e mudanças de arquivos, entre outras configurações.

"scripts": {
  "cypress:tablet-view": "cypress run --env TransferProtocol='http' --config viewportHeight=763,viewportWidth=700"
}

cypress run --config-file <configuration-file>

O Cypress pode sobrescrever o arquivo de configuração padrão localizado em /cypressRootDirectory/cypress.json.

cypress run --headed

Por padrão, o Cypress executa testes no modo headless quando o comando cypress run é usado. No entanto, se você quiser que os testes sejam executados com uma interface visual do navegador em vez de apenas a linha de comando, você pode usar a opção --headed.

cypress run --no-exit

Por padrão, o Cypress sairá do test runner após os testes terem sido executados. Passando o --no-exit para o comando cypress run, você pode garantir que o test runner permaneça aberto no terminal.

cypress run --parallel

A opção --parallel nos permite distribuir os testes Cypress em múltiplas máquinas, reduzindo assim o tempo geral para executar a suíte de testes.

cypress run --record

Ao gravar um conjunto de testes no Cypress, garantimos que os resultados dos testes, juntamente com os vídeos e screenshots tirados durante a execução do teste, são gravados no dashboard do Cypress.

cypress run --spec <spec-file>

O Cypress tem a capacidade de executar um único arquivo de especificação usando a opção --spec no momento de execução. Isso é útil quando você deseja executar um único arquivo de especificação em vez de toda a suíte de testes.

cypress open

O comando cypress open abre o test runner do Cypress em modo headed. O test runner do Cypress é uma interface gráfica do usuário que permite aos usuários ver os testes sendo executados em tempo real. As opções de configuração mais comuns para o comando cypress open incluem:

cypress open --env <env-variable>

Assim como a opção --env no cypress run, podemos passar variáveis de ambiente para o comando cypress open.

cypress open --browser <browser-path-or-name>

Assim como a opção --browser no cypress run, podemos especificar o navegador que o Cypress deve usar ao abrir o test runner.

cypress open --config <configuration-option>

Assim como no cypress run, podemos sobrescrever as configurações do Cypress quando abrimos o test runner.

cypress open --project <project-path>

O Cypress vem com a capacidade embutida de sobrescrever o caminho padrão que o Cypress inicia ao executar os testes.

"scripts": {
  "cypress:project:v3": "cypress open --env TransferProtocol='http' --project 'cypress/todo-app-v3/'"
}

cypress open --port <port-number>

Por padrão, o Cypress roda na porta 8080. Executar o comando cypress run passando a opção --port permite sobrescrever a porta padrão onde os testes são executados.

"scripts": {
  "cypress:open:changed-port": "cypress open --env TransferProtocol='http' --port 3004"
}

Usando o comando cypress info

Executar o comando cypress info em um terminal imprimirá as informações de instalação do Cypress e a configuração do ambiente no terminal. As informações impressas pelo comando incluem:

  • Navegadores instalados na máquina que foram detectados pelo Cypress.
  • Informações sobre o sistema operacional do host.
  • A localização do cache do binário do Cypress.
  • A localização do armazenamento de dados de tempo de execução.
  • Variáveis de ambiente que foram prefixadas com CYPRESS.

Usando o comando cypress version

O comando cypress version imprime a versão binária do Cypress e a versão npm do módulo Cypress que foi instalado.

Depuração do Cypress na linha de comando

Nesta seção, exploraremos como usar as propriedades de depuração da linha de comando do Cypress para solucionar os problemas que podemos encontrar ao executar nossos testes.

O Cypress tem um módulo de depuração embutido que pode ser exposto aos usuários passando comandos de depuração antes de executar os testes. Para receber saída de depuração do terminal, a variável de ambiente DEBUG precisa ser definida antes das execuções de testes Cypress em ambientes Mac ou Linux.

"scripts": {
  "cypress:open:debugger": "DEBUG=cypress:* cypress open --env TransferProtocol='http'",
  "cypress:run:debugger": "DEBUG=cypress:* cypress run --env TransferProtocol='http'"
}

Executar npm run cypress:open:debugger executará os testes Cypress em um terminal e registrará a saída de depuração da execução. O segundo comando, que pode ser executado com npm run cypress:run:debugger, executará o depurador enquanto executa os testes no test runner do Cypress.

Para filtrar a saída de depuração para módulos específicos:

"scripts": {
  "cypress:open:server-debugger": "DEBUG=cypress:server:* cypress open --env TransferProtocol='http'"
}

Escrevendo seu primeiro teste

Os testes Cypress não são diferentes de nenhum outro teste. Assim como em todos os outros testes, os testes Cypress devem passar quando o resultado esperado é consistente com o que a aplicação sob teste está retornando.

Testes com falha, aprovados e ignorados

Testes aprovados são os testes que não encontram nenhum problema durante a execução e que atendem ao resultado esperado definido no código do teste.

Testes com falha são testes que são esperados para passar, mas que encontraram problemas durante a execução, levando à falha do teste.

Testes ignorados são os testes que o desenvolvedor não quer executar em um ponto particular da execução dos testes, e são assim marcados por palavras-chave específicas que instruem o Cypress a não executar os testes marcados.

Escrevendo testes práticos

it('can visit a url', () => {
  cy.visit('http://todomvc.com/examples/react/#/')
});
it('can find a todo input element', () => {
  cy.visit('http://todomvc.com/examples/react/#/')
  cy.get(".new-todo")
});
it('can type in a todo input element', () => {
  cy.visit('http://todomvc.com/examples/react/#/')
  cy.get('.new-todo').type('New Todo {enter}')
});

Asserindo o estado da aplicação

it('asserts change in application state', () => {
  cy.visit('http://todomvc.com/examples/react/#/')
  cy.get('.new-todo').type('New Todo {enter}')
  cy.get('.new-todo').type('Another Todo {enter}')
  cy.get(".todo-list").find('li').should('have.length', 2)
});
it('asserts inserted todo items are present', () => {
  cy.visit('http://todomvc.com/examples/react/#/')
  cy.get('.new-todo').type('New Todo {enter}')
  cy.get('.new-todo').type('Another Todo {enter}')
  cy.get(".todo-list").find('li').should('have.length', 2)
  cy.get('li:nth-child(1)>div>label').should('have.text', 'New Todo')
  cy.get('li:nth-child(2)>div>label').should('have.text', 'Another Todo')
});

O recurso de auto-reload do Cypress

Por padrão, o Cypress monitora mudanças de arquivo e recarrega os testes imediatamente quando uma mudança de arquivo é detectada. Esse recurso permite que você tenha feedback instantâneo sem precisar reexecutar os testes após fazer mudanças nos arquivos de teste.

Para desabilitar o auto-reload via cypress.json:

{
  "watchForFileChanges": "false"
}

Ou via opção de linha de comando:

cypress open --config watchForFileChanges=false

Asserções Cypress

As asserções Cypress se originam dos módulos chai, chai-jquery e sinon-chai, que vêm com a instalação do Cypress. Podemos escrever asserções Cypress de duas formas: definindo sujeitos explicitamente ou implicitamente.

  • Sujeitos implícitos: .should() ou .and()
  • Sujeitos explícitos: expect()

Sujeitos implícitos

describe('Cypress Assertions', () => {
  it('Using Implicit subjects - should', () => {
    cy.visit('http://todomvc.com/examples/react/#/')
    cy.get(".new-todo").should('have.attr', 'placeholder', 'What needs to be done?')
  });
});
it('Using Implicit subjects - and()', () => {
  cy.visit('http://todomvc.com/examples/react/#/')
  cy.get(".new-todo")
    .should('have.attr', 'placeholder', 'What needs to be done?')
    .and('have.class', 'new-todo')
});

O que é TDD?

O Test-Driven Development (TDD), ou Desenvolvimento Orientado por Testes, é um processo de desenvolvimento que é dependente da repetição de ciclos de desenvolvimento curtos. O processo começa com um desenvolvedor escrevendo um teste de caso de uso que define uma melhoria desejada ou nova função, antes de escrever qualquer código de produção.

O processo de TDD segue o seguinte ciclo:

  1. Vermelhor (Red): Escrever um teste que falha.
  2. Verde (Green): Escrever o código mínimo para fazer o teste passar.
  3. Refatorar (Refactor): Refatorar o código para melhorar sua qualidade.

Capítulo 7 — Entendendo a Interação com Elementos no Cypress

Neste capítulo, aprenderemos como o Cypress interage com elementos do DOM e como ele determina a acionabilidade (actionability) dos elementos.

Abordaremos os seguintes tópicos principais neste capítulo:

  • Entendendo acionabilidade
  • Forçando acionabilidade
  • Interagindo com elementos

Entendendo acionabilidade

A acionabilidade é a capacidade do Cypress de determinar se um elemento do DOM está em um estado em que pode ser interagido. O Cypress executa verificações antes de realizar qualquer ação em um elemento, verificando:

  • Visibilidade
  • Desabilitação
  • Desanexamento do DOM
  • Modo somente leitura
  • Animações
  • Cobertura por outros elementos
  • Rolagem
  • Coordenadas

Visibilidade: O Cypress verifica se o elemento é visível no DOM verificando: - Que o elemento não tem seu CSS display definido como none - Que o elemento não tem seu CSS visibility definido como hidden - Que o elemento não tem seu CSS opacity definido como 0 - Que o elemento tem tanto largura quanto altura acima de 0

Animações: O Cypress tem um mecanismo para calcular quando um elemento está animando. Há um limite de animação padrão de 1000 ms, que pode ser configurado via cypress.json ou cypress open --config animationDistanceThreshold=50.

Cobertura por elementos pai: O Cypress verifica que o elemento não está totalmente coberto por um elemento pai como um popup, overlay ou caixa de diálogo.

Coordenadas: Após o Cypress ter concluído o processo de verificação de acionabilidade, por padrão ele dispara eventos no centro do elemento. Pode-se sobrescrever essa posição:

it('can mark a todo as completed - with changed hitbox position', () => {
  cy.visit('http://todomvc.com/examples/react/#/')
  cy.get(".new-todo").type("New Todo {Enter}");
  cy.get(".new-todo").type("Another New Todo {Enter}");
  cy.get('.todo-list>li:nth-child(1)').find('.toggle').click({ position: 'topLeft' });
});

Forçando acionabilidade

Para sobrescrever as verificações de acionabilidade do Cypress, podemos passar o parâmetro {force: true} para os comandos acionáveis do Cypress:

it('can mark all todo as completed - with forced toggle option (Success)', () => {
  cy.get(".new-todo").type("New Todo {Enter}");
  cy.get(".new-todo").type("Another New Todo {Enter}");
  cy.get('.todo-list>li:nth-child(1)').find('.toggle').click();
  cy.get('#toggle-all').click({force: true});
});

Quando a acionabilidade é forçada em um comando, o Cypress:

  • Continua a executar todas as ações padrão
  • Força o disparo do evento no elemento

Porém, o Cypress não irá:

  • Garantir que o elemento seja visível
  • Rolar o elemento para a visualização
  • Garantir que o elemento não esteja desabilitado
  • Garantir que o elemento não esteja desanexado
  • Garantir que o elemento não esteja animando
  • Garantir que o elemento não esteja coberto
  • Garantir que o elemento não seja somente leitura

Neste capítulo, aprenderemos como utilizar o Test Runner para escrever melhores testes. Abordaremos os seguintes tópicos:

  • Entendendo o painel de instrumentos
  • Entendendo o Selector Playground

Entendendo o painel de instrumentos

O painel de instrumentos é um painel especial no Test Runner do Cypress que só é visível quando o Cypress está fornecendo informações adicionais sobre seus testes. A aparência do painel de instrumentos é acionada por comandos específicos como cy.stub(), cy.intercept() e cy.spy().

Intercepts

O Cypress usa o comando cy.intercept() para gerenciar o comportamento das requisições HTTP na camada de rede de um teste.

describe('Routing a request', () => {
  it('can wait for a app initialization', () => {
    cy.intercept('POST', '**/j/**').as('initializeTodoApp');
    cy.visit('http://todomvc.com/examples/react/#/');
    cy.wait('@initializeTodoApp');
  })
});

Stubs

Stubs são usados para substituir uma função, método ou objetos de um objeto com lógica personalizada. O Cypress usa cy.stub() para criar stubs.

it('cy.stub(): stubs an object method', () => {
  const obj = {
    foo() {}
  }
  const stub = cy.stub(obj, 'foo').as('fooStub')
  obj.foo('foo', 'bar')
  expect(stub).to.be.called
  expect(stub).to.be.calledWith('foo', 'bar')
  expect(stub.args[0][0]).to.eq('foo')
  expect(stub.args[0][1]).to.eq('bar')
});

Spies

Spies fornecem a capacidade de inspecionar como uma função ou método foi chamado. Ao contrário dos stubs, os spies não alteram o comportamento da função que estão monitorando.

it('cy.spy(): wraps a method in a spy', () => {
  const obj = {
    foo() {}
  }
  const spy = cy.spy(obj, 'foo').as('fooSpy')
  obj.foo('foo', 'bar')
  expect(spy).to.be.called
  expect(spy).to.be.calledWith('foo', 'bar')
});

Entendendo o Selector Playground

O Selector Playground é uma ferramenta interativa do Test Runner do Cypress que nos fornece acesso a um seletor único para identificar elementos no DOM. Podemos usar o Selector Playground para:

  • Determinar o seletor CSS único
  • Encontrar seletores para elementos
  • Verificar se o seletor retorna os elementos corretos

Entendendo como fazer stub de requisições

Fazendo stub de requisições XHR

describe('XHR Stubbed Requests', () => {
  it('Stubs bank Account XHR server response', () => {
    cy.intercept('GET', 'bankAccounts',
      {results: [{id: "RskoB7r4Bic", userId: "t45AiwidW",
      bankName: "Test Bank Account", accountNumber: "6123387981",
      routingNumber: "851823229", isDeleted: false}]})
      .as('bankAccounts');
    cy.intercept('GET', 'transactions/public').as('transactions');
    cy.intercept('notifications').as('notifications');

    cy.wait('@transactions').its('response.statusCode').should('eq', 304);
    cy.wait('@notifications').its('response.statusCode').should('eq', 304);
    cy.wait('@bankAccounts').then((xhr) => {
      expect(xhr.response.body.results[0].bankName).to.eq('Test Bank Account')
      expect(xhr.response.body.results[0].accountNumber).to.eq('6123387981')
    });
  });
});

Vantagens do stubbing de requisições:

  • Ter controle sobre o corpo, cabeçalhos e status das respostas.
  • Tempos de resposta rápidos.
  • Nenhuma mudança de código necessária no servidor.
  • Simulações de atraso de rede podem ser adicionadas à requisição.

Desvantagens do stubbing de requisições:

  • A incapacidade de ter cobertura de testes em alguns endpoints do servidor.
  • Nenhuma garantia de que os dados de resposta e os dados de stub sejam iguais.

Entendendo como espionar métodos em um teste

Os spies obtêm o estado do método ou requisição e não têm a capacidade de modificá-los.

it('cy.spy(): calls sum method with arguments', () => {

      return a + b
    }
  }
  const spyRequest = cy.spy(obj, 'sum').as('sumSpy');
  const spyRequestWithArgs = spyRequest.withArgs(1, 2).as('sumSpyWithArgs')
  obj.sum(1, 2); // gatilho do spy
  expect(spyRequest).to.be.called;
  expect(spyRequestWithArgs).to.be.called;

Capítulo 12 — Testes Visuais no Cypress

Neste capítulo, aprenderemos sobre testes visuais e sua importância, bem como os viewports e ferramentas que podemos usar para realizar testes visuais automatizados.

Testes visuais

Os testes visuais são a capacidade de verificar aspectos visuais da interface de usuário de uma aplicação e garantir que eles sejam consistentes com as expectativas.

Por que fazer testes visuais?

Os testes visuais existem para dar a desenvolvedores e testadores a confiança de que a interface de usuário de uma página web não foi quebrada por nenhuma mudança do desenvolvedor. Existem dois tipos de testes visuais:

  • Testes visuais manuais
  • Testes visuais automatizados

Testes visuais manuais: Envolvem usar os olhos para validar se as mudanças feitas pela equipe de desenvolvimento não quebraram nenhuma funcionalidade visual da interface.

Testes visuais automatizados: Envolvem testar os elementos visuais da página usando um processo automatizado. Para executar testes visuais automatizados adequadamente, devemos salvar e definir nossa interface de usuário desejada como uma linha de base (baseline).

Os dois tipos de testes visuais automatizados são:

  1. Testes baseados em comparação pixel a pixel: Uma linha de base da aplicação é tirada e usada como ponto de referência para identificar mudanças na interface de usuário ao longo do tempo.
  2. Testes baseados em comparação por IA: Uma linha de base é obtida e usada como referência. Ao contrário da comparação pixel a pixel, a comparação baseada em IA usa IA para identificar mudanças na interface de usuário.

Viewports no Cypress

Os viewports são as dimensões da área de visualização de uma janela do navegador, e podem ser configurados no Cypress de várias formas, incluindo via cypress.json, especificando o viewport no arquivo de teste ou passando o viewport como um argumento de linha de comando.

{
  "viewportHeight": 568,
  "viewportWidth": 320
}
describe('Cypress Viewport tests', () => {
  it('changes to a 340x480 viewport', () => {
    cy.viewport(340, 480)
  })

  it('shows the default viewport', () => {
    cy.viewport('macbook-15')
  })
})

Percy

  • Integração fácil com pipelines de CI/CD
  • Revisão visual colaborativa
  • Gerenciamento de aprovação de screenshot

O Applitools Eyes é uma ferramenta de testes visuais alimentada por IA que usa algoritmos de Inteligência Artificial para comparar screenshots de testes com screenshots de linha de base.

O Applitools usa três comandos para controlar os testes Cypress:

  1. cy.eyesOpen(): para inicializar e iniciar o teste
  2. cy.eyesCheckWindow(): responsável por tirar screenshots
  3. eyesClose(): conclui a sessão do Applitools Eyes e carrega as screenshots para o dashboard
it('applitools: can signin on login page - second build snapshot', () => {
  cy.eyesOpen({
    appName: 'SignIn Page',
    browser: { width: 1200, height: 720 },
  });

  cy.get('#username').type('Kat');
  cy.get('#password').type('passd');
  cy.eyesCheckWindow('loginPage');

  cy.eyesClose();
});

Nota importante Lembre-se de adicionar sua variável APPLITOOLS_API_KEY, que foi obtida no dashboard do Applitools, à sua máquina local como uma variável de ambiente antes de executar seus testes. Este token garante que os snapshots tirados pelo Applitools Eyes SDK sejam carregados com sucesso no seu dashboard do Applitools.
